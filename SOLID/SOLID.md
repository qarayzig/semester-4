# Задание по SOLID

## Выбранный принцип

Для примера был выбран принцип **SRP** — Single Responsibility Principle, или принцип единственной ответственности.

Суть принципа заключается в том, что у класса должна быть только одна причина для изменения. То есть класс должен отвечать только за одну задачу.

---

## Предметная область

В качестве предметной области рассмотрим интернет-магазин.

Допустим, есть заказ покупателя. Необходимо:

- хранить товары в заказе;
- считать итоговую стоимость;
- сохранять заказ в файл;
- отправлять уведомление покупателю.

---

## Пример с нарушением принципа SRP

В данном примере один класс `Order` выполняет сразу несколько разных задач.

```python
class Order:
    """Класс заказа, нарушающий принцип единственной ответственности."""

    def __init__(self, items):
        """Инициализирует заказ списком товаров."""
        self.items = items

    def calculate_total(self):
        """Считает итоговую стоимость заказа."""
        return sum(item["price"] for item in self.items)

    def save_to_file(self):
        """Сохраняет заказ в файл."""
        with open("order.txt", "w", encoding="utf-8") as file:
            file.write(str(self.items))

    def send_email(self):
        """Отправляет уведомление покупателю."""
        print("Письмо покупателю отправлено")
```

---

## В чем нарушение

Класс `Order` отвечает сразу за несколько обязанностей:

1. хранение данных о заказе;
2. расчет стоимости;
3. сохранение заказа в файл;
4. отправку уведомления.

Это нарушает принцип единственной ответственности.

Если изменится способ сохранения заказа, придется менять класс `Order`.

Если изменится способ отправки уведомлений, снова придется менять класс `Order`.

Если изменится логика расчета стоимости, опять придется менять тот же самый класс.

Таким образом, у класса слишком много причин для изменения.

---

## Исправленный вариант

Разделим обязанности между несколькими классами.

```python
class Order:
    """Класс, отвечающий только за хранение данных заказа."""

    def __init__(self, items):
        """Инициализирует заказ списком товаров."""
        self.items = items


class OrderCalculator:
    """Класс, отвечающий за расчет стоимости заказа."""

    def calculate_total(self, order):
        """Считает итоговую стоимость заказа."""
        return sum(item["price"] for item in order.items)


class OrderRepository:
    """Класс, отвечающий за сохранение заказа."""

    def save_to_file(self, order):
        """Сохраняет заказ в файл."""
        with open("order.txt", "w", encoding="utf-8") as file:
            file.write(str(order.items))


class EmailNotifier:
    """Класс, отвечающий за отправку уведомлений."""

    def send_email(self):
        """Отправляет уведомление покупателю."""
        print("Письмо покупателю отправлено")
```

---

## Пример использования исправленного кода

```python
items = [
    {"name": "Ноутбук", "price": 50000},
    {"name": "Мышь", "price": 1500},
]

order = Order(items)

calculator = OrderCalculator()
repository = OrderRepository()
notifier = EmailNotifier()

total = calculator.calculate_total(order)
repository.save_to_file(order)
notifier.send_email()

print(f"Итоговая стоимость заказа: {total}")
```

---

## Комментарий к исправлению

После исправления каждый класс отвечает только за одну задачу.

Класс `Order` хранит информацию о заказе.

Класс `OrderCalculator` отвечает только за расчет итоговой стоимости.

Класс `OrderRepository` отвечает только за сохранение заказа.

Класс `EmailNotifier` отвечает только за отправку уведомлений.

Теперь при изменении логики отправки письма нужно будет изменить только `EmailNotifier`.

При изменении способа сохранения заказа нужно будет изменить только `OrderRepository`.

При изменении расчета стоимости нужно будет изменить только `OrderCalculator`.

Таким образом, код стал более понятным, гибким и удобным для сопровождения.

---

## Вывод

В данном примере был рассмотрен принцип единственной ответственности из SOLID.

Сначала был создан класс, который нарушал этот принцип, так как выполнял несколько разных задач одновременно.

Затем обязанности были разделены между несколькими классами. После исправления каждый класс стал иметь только одну причину для изменения, что соответствует принципу SRP.
